## Problem Statement

Build a macOS app with Widget support for people who set long-term goals and want a very low-friction way to confirm daily progress.

The product is not a general TODO app, not a checklist manager, and not a streak-maximization habit tracker. It is a local-first tool for answering one daily question: did I continue my long-term commitment today?

Example target use case: "I want to study vocabulary for 30 minutes every day for 200 days." The app should let the user create that commitment, put it on the desktop as a Widget, and mark today's completion with one click.

## What Makes This Cool

The cool version is that the Widget is the product. The user does not open an app every day, browse a task list, manage categories, or fight a dashboard. They glance at the desktop, click the specific goal they completed, and move on.

The emotional stance matters: this app should not shame missed days. If a day is missed, it is missed. The point is not to preserve a fake streak. The point is to keep an honest record that says, "I am still on the path."

## Constraints

- macOS app with Widget support.
- Widget must support direct same-day check-in without opening the main app.
- Users can create any number of long-term goals.
- Widget cannot show unlimited goals, so it displays the first N goals based on user-defined ordering and Widget size.
- MVP should avoid reminders, tags, notes, social features, cloud sync, and complex task hierarchies.
- First version should be local-first and open-source-friendly.

## Premises

1. This is a long-term goal daily confirmation tool, not a normal TODO or checklist app.
2. Widget is the core product interface. The main app is a configuration and review center.
3. Users can have many goals, but Widget display is size-limited and follows the user's manual ordering.
4. Missed days should be represented honestly. No make-up check-ins, no streak repair, no guilt mechanics.
5. MVP should stay narrow: no reminders, labels, notes, cloud sync, social sharing, or subtask trees.

## Approaches Considered

### Approach A: Widget-First MVP

A fast first version with multi-goal creation, user ordering, Widget display, one-click daily check-in, and a simple progress view.

Effort: M  
Risk: Low-Med

Pros:
- Ships the core daily workflow quickly.
- Tests whether Widget-first check-in feels good.
- Keeps the product boundary tight.

Cons:
- Progress analytics are intentionally simple.
- Data portability and future sync need careful follow-up design.
- Product personality may feel plain if the dashboard is too bare.

### Approach B: Local-First Open Source Core

Start by designing a strong local data layer with import/export, stable storage format, migration path, and clean separation between app, Widget, and persistence.

Effort: L  
Risk: Med

Pros:
- Strong open-source foundation.
- Easier future support for iCloud, backup, CLI tools, or external automation.
- Data ownership is explicit from day one.

Cons:
- Slower path to a usable daily product.
- Risks spending too much time on storage design before validating the interaction.
- Can pull attention away from the Widget experience.

### Approach C: Still On The Path Experience

Build the product around the emotional model of honest long-term progress instead of streak preservation. The dashboard emphasizes recent progress, accumulated effort, and forward motion.

Effort: M-L  
Risk: Med

Pros:
- Strongest differentiation from common habit trackers.
- Matches the user's stated philosophy: "we don't need to complete tasks just to preserve a streak; we want to know we are still moving forward."
- Creates a clearer product voice.

Cons:
- Requires careful copy and visual design.
- If overdone, it can become sentimental or distracting.
- Needs a strict MVP boundary to avoid becoming a journaling app.

### Approach D: Widget-First Long-Term Goals

A combined product direction: technical scope from Approach A, product philosophy from Approach C, with enough local-first structure to keep the open-source path clean.

The main app lets users create, edit, delete, and sort long-term goals. Each goal has a name, daily action, target day count, start date, order, and daily completion records. The Widget adapts to its size and shows the first N goals according to user ordering. Each displayed goal has a direct check-in control for today's completion.

Effort: M  
Risk: Med

Pros:
- Best match for the intended product.
- Makes the Widget the primary workflow rather than an accessory.
- Keeps the first version focused while preserving a clean path to future local-first features.
- Differentiates from bloated habit trackers by refusing unnecessary management features.

Cons:
- Widget interaction and shared persistence must be designed correctly from the start.
- Dashboard scope needs discipline.
- Some users may expect reminders or sync, and the product must be comfortable saying "not in v1."

## Recommended Approach

Choose Approach D: Widget-First Long-Term Goals.

This is the clearest version of the product. It does not try to compete with Things, Reminders, TickTick, Habitify, or Notion. It does one job: help a user confirm daily progress on long-term commitments from the desktop.

## Product Model

### Goal

A long-term commitment the user wants to pursue for a defined number of days.

Fields:
- Title: "Vocabulary"
- Daily action: "Study 30 minutes"
- Target completion days: 200
- Start date
- Manual sort order
- Active / archived state

MVP semantics:
- Target days means intended completed days, not calendar duration.
- Progress is completed records / target completion days.
- Missing a day does not reduce the target or create a repair flow. It simply means there is no completion record for that date.
- Start date is locked after the first completion.
- Title and daily action can be edited after history exists.
- Target completion days can be increased, or decreased only down to the current completed count.

### Daily Record

A dated record of whether the goal was completed on that day.

Fields:
- Goal ID
- Date
- Completed boolean
- Completion timestamp

Rules:
- A goal can be completed once per day.
- Tapping an already completed goal in the Widget does nothing in MVP.
- Undo is available only in the main app, where the user has more context.
- Missed days remain missed. No backfilling in MVP.

### Widget Display

Widget content follows the user's ordering and adapts by size. MVP rule: show active goals by manual order only. There is no separate Widget pinning list in v1.

Suggested behavior:
- Small Widget: show 1 goal.
- Medium Widget: show 2-4 goals depending on layout.
- Large Widget: show more goals, with compact progress context.

Each visible goal shows:
- Goal title
- Daily action
- Today's completion state
- Minimal progress indicator, such as completed days / target days
- A direct check-in button or toggle

### Dashboard

The dashboard is for occasional review, not daily operation.

It should show:
- Today status across active goals
- Goal list with progress
- Completed days / target days
- Remaining days
- Completion rate
- Recent 30-day activity
- Manual sort controls

It should not show:
- Complex productivity analytics
- Categories and tags
- Notes and journaling
- Streak repair
- Gamified rewards
- Social comparison

### First-Run Flow

The first launch should make the Widget-first path obvious without turning the app into an onboarding product.

Suggested flow:
- Empty state invites the user to create one long-term goal.
- After the first goal is saved, the app briefly points the user toward adding the One Step Widget.
- The app remains useful without the Widget, but the primary success path is: create goal -> add Widget -> complete from desktop.
- Avoid multi-step tutorials, sample data, accounts, permissions prompts unrelated to Widgets, or reminder setup in v1.

## Technical Direction

Use a native macOS architecture:

- SwiftUI for the main app.
- WidgetKit for desktop Widgets.
- AppIntent for direct Widget check-in actions.
- App Group shared storage so the main app and Widget extension can read and write the same records.
- SwiftData for MVP persistence, stored in an App Group container shared by the main app, Widget extension, and AppIntent code.
- A small repository layer should own reads and writes so Widget and app code do not duplicate persistence details.
- The repository should use calendar-day keys normalized to the user's current calendar and local date.
- Goal identifiers should be stable UUIDs, and daily completion uniqueness should be enforced by goal/day semantics.
- Keep the persistence model migration-friendly even before import/export exists. Do not rely on UI-only assumptions for data integrity.

Important implementation constraint: the Widget action must update shared data and trigger Widget refresh so the completed state appears quickly after the click. Prototype this AppIntent + App Group write path before building the full dashboard.

## MVP Decisions

1. Completed goals are not undoable from the Widget in MVP. Undo lives in the main app.
2. Archived goals remain available in historical dashboard views, but do not appear in Widgets.
3. Target days means intended completion days, not calendar duration.
4. Editing after history is allowed for title and daily action. Start date is locked after the first completion. Target completion days can be increased, or decreased only down to completed count.
5. Widget shows active goals by manual order only. Separate Widget pinning is deferred.

## Success Criteria

The first version is successful if:

- A user can create at least 5 long-term goals.
- The user can manually reorder goals.
- A Widget displays active goals according to that order and adapts to size.
- The user can complete today's check-in directly from the Widget without opening the app.
- Widget state refreshes after check-in without requiring the user to manually open the app.
- The main app shows accurate progress and recent history.
- Missed days are represented honestly without streak repair mechanics.
- The product remains simple enough that a new user understands it in under 60 seconds.

## Verification Plan

Core tests:
- Goal progress uses completed record count, not elapsed calendar days.
- One goal can be completed only once per local day.
- Manual ordering determines Widget order.
- Archived goals are excluded from Widget data.
- Target completion days cannot be set below completed count.

Widget/AppIntent verification:
- Manual test: create 5 goals, add the Widget, click a visible incomplete goal, confirm it changes to completed without opening the app.
- Manual test: change goal order in the app, reload Widget timeline, confirm displayed order changes.
- Manual test: use small, medium, and large Widget sizes and confirm each shows the expected number of goals without clipping.
- Integration test where possible: AppIntent writes a completion record to the App Group store and repository reads it back.

## Distribution Plan

Because this is an open-source macOS app, the project should plan for distribution early.

MVP distribution:
- Source available on GitHub.
- Developer build instructions in README.
- Local developer build first. GitHub Releases wait until app + Widget are working reliably.

Later distribution:
- Optional notarized builds.
- Optional Homebrew Cask.
- Optional Mac App Store only if sandboxing, review, and distribution tradeoffs are worth it.

CI/CD:
- GitHub Actions should eventually build the app and run tests on pull requests.
- Release workflow can package the app and attach artifacts to GitHub Releases.
- Signing and notarization can be deferred until the core product works locally.
