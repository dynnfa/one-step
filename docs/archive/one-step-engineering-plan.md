## Goal

Build the first vertical slice for One Step: a macOS 14+ SwiftUI app with a Widget extension where a user-created long-term goal can be completed directly from the Widget and reflected in shared local state.

This is not the full MVP. It proves the highest-risk product path first.

## Architecture Decisions

### 1. Scope: Vertical Slice First

Decision: 1A, vertical slice first.

The first implementation must prove:

```text
Widget Button
    │
    ▼
CompleteGoalIntent.perform()
    │
    ▼
GoalRepository.completeToday(goalID, LocalDay.today)
    │
    ▼
SwiftData store in App Group container
    │
    ▼
WidgetCenter reload
    │
    ▼
Widget shows completed state
```

Do not build full dashboard, reminders, import/export, iCloud sync, or drag sorting in this slice.

Vertical slice boundary:
- Build enough main app UI to create goals and confirm repository state.
- Build enough Widget UI to prove same-day completion from the desktop.
- Do not implement the full MVP dashboard, manual drag sorting UI, archived history browsing, import/export, or distribution packaging in this slice.
- Keep model fields that future MVP work needs, such as `sortOrder` and `archivedAt`, even when the first UI does not expose every operation.

### 2. Platform: macOS 14+

Decision: 2A, macOS 14+ only.

Use SwiftData, WidgetKit interactive widgets, and AppIntent directly. No macOS 13 fallback path in v1.

### 3. Shared Core Boundary

Decision: 3A, create a shared `OneStepCore` local Swift package/module.

`OneStepCore` owns:
- SwiftData models
- Repository
- Date normalization
- Snapshot structs
- Domain errors
- Unit tests

The app and Widget depend on `OneStepCore`. `OneStepCore` must not depend on SwiftUI views or WidgetKit.

```text
OneStep.app ───────────────┐
                           ▼
                     OneStepCore
                           ▲
OneStepWidgetExtension ────┘
```

### 4. Persistence Shape

Decision: 4A, SwiftData models behind a repository returning plain snapshots.

Rules:
- SwiftData model types stay inside `OneStepCore`.
- SwiftUI app views and Widget views render snapshot structs.
- Repository owns all SwiftData reads and writes.
- Use an App Group container shared by app, Widget extension, and AppIntent code.
- Goal IDs must be stable UUIDs.
- Daily completions must be unique by `goalID + dayKey`; duplicate same-day writes are repository-level no-ops.
- Keep model names and fields migration-friendly. Add an explicit schema/version note before the first public release.

### 5. Date Semantics

Decision: 5A, store normalized `LocalDay` key plus completion timestamp.

`DailyCompletion` stores:
- `dayKey`, for example `2026-04-29`
- `completedAt`, the actual timestamp

Rules:
- One completion per goal per local day.
- `LocalDay` is computed from the user's current calendar/timezone at completion time.
- Progress uses count of completion records, not elapsed calendar days.

### 6. Vertical Slice UI

Decision: 6A, add goal form and list, no drag sorting yet.

The app should include:
- Goal list
- Add goal form
- Basic completed / remaining count
- A simple first-run empty state that points the user toward creating the first goal and adding the Widget after creation

The model should include `sortOrder` now. For this slice, default `sortOrder` to creation order. Manual drag sorting is deferred.

Widget target behavior:
- Small Widget: 1 goal
- Medium Widget: 3 goals
- Large Widget: 5 goals

### 8. Repository Shape

Use one concrete repository first. Do not add protocol abstractions until a real second implementation exists.

Expected repository surface:

```swift
public struct GoalRepository {
    public func createGoal(...)
    public func activeGoalsForWidget(limit: Int, day: LocalDay) throws -> [WidgetGoalSnapshot]
    public func goalsForList(day: LocalDay) throws -> [GoalListSnapshot]
    public func completeToday(goalID: Goal.ID, day: LocalDay) throws
}
```

### 9. Error Handling

Decision: 9A, typed domain errors plus logging.

Expected domain errors:

```swift
public enum GoalRepositoryError: Error {
    case goalNotFound
    case goalNotActive
    case invalidTargetCompletionDays
    case targetBelowCompletedCount
    case storeUnavailable
    case saveFailed
}
```

`CompleteGoalIntent.perform()` should catch repository errors and log via `os.Logger`. Widget failures should not silently vanish during development.

### 10. App Group + AppIntent Spike Acceptance

Before expanding the app UI, prove the shared write path in the smallest possible build.

Acceptance checklist:
- App and Widget extension both have the same App Group entitlement.
- `OneStepCore` can create a SwiftData `ModelContainer` whose store URL lives in the App Group container.
- The main app can create a goal and the Widget timeline provider can read it from the same store.
- `CompleteGoalIntent.perform()` can write a completion through `GoalRepository`.
- After the intent runs, `WidgetCenter` reload is requested and the Widget eventually renders the completed state.
- Development failures log enough detail to distinguish entitlement, store creation, repository, and timeline refresh problems.

## Test Decisions

### 11. Archived Goal Stale Widget Tap

Decision: 11A, archived goals cannot be completed.

If a stale Widget timeline sends a tap for an archived goal:

```text
CompleteGoalIntent.perform()
    │
    ▼
GoalRepository.completeToday()
    │
    └── throws .goalNotActive
            │
            ▼
AppIntent logs and no-ops
```

This prevents hidden mutation of archived goals.

### 12. Concurrent Writes

The repository is the source of truth for consistency. UI state is advisory only.

Rules:
- A Widget tap and app action for the same goal/day must result in one completion record.
- A repeated Widget tap on an already completed goal is a no-op, not a user-visible failure.
- AppIntent handles repository errors by logging and returning without mutating unrelated state.

## Implementation Order

1. Scaffold `OneStep.xcodeproj` with macOS app target and Widget extension target.
2. Add `Packages/OneStepCore` local Swift package.
3. Define `Goal`, `DailyCompletion`, `LocalDay`, snapshot structs, and `GoalRepositoryError` in `OneStepCore`.
4. Configure SwiftData `ModelContainer` for App Group storage.
5. Run the App Group + AppIntent spike against the shared store before expanding UI.
6. Implement `GoalRepository` with `createGoal`, `goalsForList`, `activeGoalsForWidget`, and `completeToday`.
7. Add `OneStepCoreTests` using an in-memory SwiftData container.
8. Build app goal list and add goal form against repository snapshots.
9. Build Widget timeline provider using `activeGoalsForWidget(limit:day)`.
10. Build `CompleteGoalIntent.perform()` using `completeToday` and Widget reload.
11. Run manual Widget verification for small, medium, and large families.

## NOT in Scope

- Full dashboard, deferred until Widget check-in path works.
- Drag sorting UI, deferred; `sortOrder` exists and defaults to creation order.
- Reminders, deferred because they are not needed to prove Widget-first value.
- Tags, notes, and subtasks, explicitly out of v1 scope.
- Import/export, deferred to v2 open-source data portability.
- iCloud sync, deferred until local model is stable.
- GitHub Releases, signing, notarization, Homebrew, and Mac App Store distribution, deferred until local app + Widget are reliable.

## Failure Modes To Cover

| Flow | Failure | Handling | Test |
|------|---------|----------|------|
| Create goal | Target days <= 0 | throw `invalidTargetCompletionDays`, show app error | unit + UI/manual |
| Complete today | Goal missing from stale Widget | throw/log `goalNotFound`, no-op | unit |
| Complete today | Goal archived from stale Widget | throw/log `goalNotActive`, no-op | unit |
| Complete today | Duplicate same-day tap | idempotent no-op | unit |
| Complete today | Widget and app write same goal/day close together | one completion record | unit where feasible + manual |
| Store setup | App Group entitlement or container URL missing | throw/log `storeUnavailable` | manual spike |
| Widget reload | Timeline does not refresh immediately | manual verification, avoid promising instant OS behavior | manual |
| Date handling | Midnight/timezone boundary | `LocalDay` tests | unit |
| Widget display | Too many active goals | repository limit and Widget family cap | unit + manual |
